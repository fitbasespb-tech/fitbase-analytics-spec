## 1. Цель и контекст

### 1.1. Бизнес-контекст

FitBase — B2B SaaS CRM-система для фитнес-бизнеса (клубы, студии, школы, бассейны, падел-корты и др.). Подписочная модель, годовая подписка. Средний чек ~100 000 ₽/год. Три основных рынка: Россия, Казахстан, Узбекистан.

Подробный профиль компании: [Strategic/Knowledge/company_profile.md](../../Knowledge/company_profile.md)

### 1.2. Цели на 2026

| Метрика | Значение |
|---------|----------|
| Рост | x2 |
| Подключения (всего) | 3 000 |
| Новых из маркетинга | 1 500 |
| Выручка | 300 млн ₽ |
| MQL | ~4 600/год (~385/мес) |
| Конверсия MQL → покупка | 25% |

### 1.3. Назначение системы

Заменить Rick.ai собственной сквозной аналитикой, которая:

- связывает рекламные расходы по всем каналам и странам с лидами, MQL, SQL и оплатами в AmoCRM;
- даёт план-факт по каналам и странам;
- строит когорты по лидам (время от попадания в AmoCRM до покупки);
- прогнозирует оплаты на основе текущего пайплайна и исторических конверсий;
- отслеживает мультифилиальность (один аккаунт → несколько сделок);
- обеспечивает глубину до креатива (платные), до статьи (SEO), до площадки (посевы).

### 1.4. Основные KPI

**Первичные:** целевые лиды (MQL/SQL) и продажи (подключения).

**Параметры бизнеса:**

### 2.2. Выбор хранилища

| Вариант | Плюсы | Минусы | Рекомендация |
|---------|-------|--------|--------------|
| **ClickHouse (Yandex Cloud)** | Нативная интеграция с DataLens, быстрые аналитические запросы, колоночное хранение | Требует Yandex Cloud, сложнее для OLTP-запросов | **Приоритет** |
| PostgreSQL | Простой, универсальный, команда знает | Медленнее на аналитических запросах при росте данных | Запасной вариант |

При объёме ~1 000 лидов/мес + ~50 000 строк расходов/год — оба варианта справятся. ClickHouse выгоднее при масштабировании и нативной связке с DataLens.

### 2.3. ETL Layer

**Паттерн:** Python-скрипты по расписанию (cron / Yandex Cloud Functions).

**Частота обновления:**

| Источник | Частота | Метод |
|----------|---------|-------|
| AmoCRM | Каждые 15 мин (webhook) или 1 раз/час (polling) | API v4 |
| Рекламные кабинеты | 1 раз/день (06:00 МСК) | API каждой платформы |
| Яндекс.Метрика | 1 раз/день (07:00 МСК) | Reporting API |
| Google Analytics | 1 раз/день (07:00 МСК) | Data API v1 |
| BotHelp | 1 раз/день | API |
| Ручной ввод (посевы, ивенты) | По факту | Google Sheets → ETL |

**Webhook от AmoCRM:** при смене стадии / создании сделки → мгновенное обновление данных о лидах.

### 2.4. Инфраструктура

| Компонент | Размещение | Стоимость (оценка) |
|-----------|------------|-------------------|
| ClickHouse | Yandex Cloud (s2.micro) | ~3 000–5 000 ₽/мес |
| ETL-скрипты | Yandex Cloud Functions или VPS | ~1 000–2 000 ₽/мес |
| DataLens | Yandex Cloud (бесплатный тариф) | 0 ₽ |
| **Итого** | | **~4 000–7 000 ₽/мес** |

### 3.3. Расчётные метрики (views / materialized views)

| Метрика | Формула | Уровень |
|---------|---------|---------|
| CPL (Cost per Lead) | ad_costs.spend / COUNT(leads) | Канал × период |
| CPA_MQL | ad_costs.spend / COUNT(leads WHERE lead_type='MQL') | Канал × период |
| CPA_SQL | ad_costs.spend / COUNT(leads WHERE lead_type='SQL') | Канал × период |
| CAC | ad_costs.spend / COUNT(leads WHERE is_won=true) | Канал × период |
| Revenue per MQL | SUM(deal_value WHERE is_won) / COUNT(MQL) | Канал × период |
| Pipeline Velocity | (COUNT(MQL) × CR_won × AVG(deal_value)) / AVG(cycle_days) | Канал × период |
| CR MQL→SQL | COUNT(SQL) / COUNT(MQL) | Канал × период |
| CR SQL→Won | COUNT(won) / COUNT(SQL) | Канал × период |
| Median Stage Time | MEDIAN(time_in_stage_hours) | Стадия × канал |
| Net Revenue per Lead | SUM(account.total_value) / COUNT(DISTINCT lead_id per account) | Канал × период |

## 4. UTM-конвенция

### 4.1. Формат

Все рекламные ссылки FitBase используют единый формат UTM-разметки. Гео кодируется в `utm_campaign`, платформа — в `utm_source`.

| Параметр | Формат | Правила |
|----------|--------|---------|
| `utm_source` | Платформа, lowercase | `yandex`, `google`, `meta`, `vk`, `telegram`, `tiktok`, `youtube`, `seo`, `smm`, `seeding`, `partner`, `webinar`, `email`, `bothelp`, `direct` (прямые обращения) |
| `utm_medium` | Тип трафика, lowercase | `cpc`, `cpm`, `social`, `organic`, `seeding`, `email`, `referral`, `bot`, `event`, `partner` |
| `utm_campaign` | `{GEO}_{описание}` | Гео: `RF`, `KZ`, `UZ`. Описание: латиница, underscores. Пример: `RF_poisk_kommercheskie` |
| `utm_content` | Креатив/объявление | Уникальный идентификатор креатива. Пример: `banner-v2-studio`, `video-testimonial-01` |
| `utm_term` | Ключевое слово | Для поисковых кампаний. Пример: `crm для фитнес клуба` |

### 4.2. Маппинг каналов

| utm_source | utm_medium | Канал в дашборде | Страна |
|------------|------------|------------------|--------|
| `yandex` | `cpc` | Яндекс.Директ | РФ |
| `google` | `cpc` | Google Ads | КЗ, УЗ |
| `meta` | `cpc` / `cpm` | Meta (FB + IG) | КЗ, УЗ |
| `vk` | `cpc` / `cpm` | ВКонтакте Ads | РФ |
| `telegram` | `cpc` | Telegram Ads | РФ |
| `tiktok` | `cpc` / `cpm` | TikTok Ads | КЗ, УЗ |
| `youtube` | `cpc` / `cpm` | YouTube Ads | КЗ, УЗ |
| `seo` | `organic` | SEO / Органика | Все |
| `smm` | `social` | Соцсети (органика) | Все |
| `seeding` | `seeding` | Посевы в Telegram | РФ |
| `partner` | `referral` | Партнёрка | Все |
| `webinar` | `event` | Вебинары | Все |
| `email` | `email` | Email-рассылки | Все |
| `bothelp` | `bot` | BotHelp | Все |
| `direct` | `none` | Прямые обращения | Все |

### 4.3. Примеры полных ссылок

**Яндекс.Директ (РФ, поиск, коммерческие):**
```
https://fitbase.io/?utm_source=yandex&utm_medium=cpc&utm_campaign=RF_poisk_kommercheskie&utm_content=text-ad-v3&utm_term={keyword}
```

**Meta (КЗ, лидформа, студии):**
```
https://fitbase.tech/?utm_source=meta&utm_medium=cpc&utm_campaign=KZ_leadform_studio&utm_content=video-testimonial-01
```

**Посев в Telegram (РФ):**
```
https://t.me/fitbase_bot?start=utm_source=seeding&utm_medium=seeding&utm_campaign=RF_fitness_kanal_100k&utm_content=post-2026-03-05
```

### 4.4. Миграция с текущей разметки

Текущее состояние: utm_source используется непоследовательно (примеры: `yandex_fitbase`, `SMM`, `Bloggers`). Необходимо:

1. Составить таблицу маппинга старых значений → новые (для исторических данных).
2. Обновить UTM во всех активных рекламных кампаниях.
3. В ETL-слое реализовать нормализацию: `yandex_fitbase` → `yandex`, `SMM` → `smm`, `Bloggers` → `seeding`.

## 5. Интеграции

### 5.1. AmoCRM API

**Роль:** Центральный источник данных о лидах, сделках, воронках.

| Параметр | Значение |
|----------|----------|
| API версия | v4 |
| Авторизация | OAuth 2.0 (токен уже есть, MCP-подключение работает) |
| Частота синхронизации | Webhook (мгновенно) + полная синхронизация 1 раз/час |
| Существующий код | `amocrm-analytics/etl/sync.py` |

**Данные для выгрузки:**

| Эндпоинт | Данные | Поля |
|----------|--------|------|
| `/api/v4/leads` | Сделки | id, name, price (Бюджет), status_id, pipeline_id, created_at, closed_at, loss_reason, custom_fields (Тип заявки, UTM), tags, responsible_user_id |
| `/api/v4/leads/{id}` | Детали сделки | То же + embedded contacts |
| `/api/v4/contacts` | Контакты | id, name, company_id, custom_fields (телефон, email) |

## 7. Методология прогноза

### 7.1. Pipeline-based Forecast

**Логика:** Текущие сделки на каждой стадии × исторический CR стадии→Won = прогноз оплат.

**Формула:**

```
Forecast_pipeline = Σ (deals_on_stage_i × CR_stage_i_to_won)
```

Где `CR_stage_i_to_won` — процент сделок, дошедших до оплаты из стадии i (рассчитывается по данным за последние 6 месяцев).

**Данные:**
- `stage_history` — переходы по стадиям
- `leads` — текущая стадия и сумма

**Пример расчёта:**

| Стадия | Сделок | Историч. CR→Won | Прогноз оплат | Прогноз выручки |
|--------|--------|-----------------|---------------|-----------------|
| Квалифицирован | 51 | 25% | 13 | 1.3M |
| Встреча назначена | 25 | 40% | 10 | 1.0M |
| Встреча / КП | 81 | 50% | 41 | 4.1M |
| Возражения | 8 | 70% | 6 | 0.6M |
| Ожидаем оплаты | 11 | 90% | 10 | 1.0M |
| **Итого** | | | **80** | **8.0M** |

**Горизонт:** 1–4 недели. **Точность:** высокая после 2–3 месяцев накопления данных.

### 7.2. Cohort-based Forecast

**Логика:** Лиды текущего месяца конвертируются по тому же паттерну, что и предыдущие когорты.

**Формула:**

```
Forecast_cohort_month_M =
  Leads_month_(M-2) × CR_cohort_age_60d  (уже реализовались)
+ Leads_month_(M-1) × CR_cohort_age_30d  (частично реализовались)
+ Leads_month_M     × CR_cohort_age_0-15d (ещё рано, только начали)
```

CR рассчитывается по историческим когортам с разбивкой по каналу.

**Горизонт:** 30–60 дней. **Точность:** средняя, растёт с накоплением данных.

### 7.3. Trend-based Forecast

**Логика:** Экстраполяция текущего темпа на остаток месяца.

**Формула:**

```
Forecast_trend = (Won_MTD / days_passed) × days_in_month
```

Вариант с 7-day rolling average:

```
Forecast_trend_7d = Won_last_7d / 7 × days_remaining + Won_MTD
```

**Горизонт:** до конца месяца. **Точность:** грубая, но полезна для ранних алертов (если темп < 80% от плана на 10-й день месяца → сигнал).

### 7.4. Комбинированный прогноз

На дашборде показываются все три значения:

| Тип | Значение | Уверенность |
|-----|----------|-------------|
| Pipeline-based | 80 оплат / 8.0M | Высокая |
| Cohort-based | 70 оплат / 7.0M | Средняя |
| Trend-based | 65 оплат / 6.5M | Низкая |
| **План** | **75 оплат / 7.5M** | — |

Рекомендация: использовать Pipeline-based как основной, Cohort-based для верификации, Trend-based для ранних алертов.

## 8. Account-Level Tracking

### 8.1. Проблема

Стандартная сквозная аналитика считает: 1 лид = 1 сделка. У FitBase один аккаунт (сеть/франшиза) может подключить несколько филиалов = несколько сделок. Без account-level tracking каналы, приводящие сетевых клиентов, недооцениваются.

### 8.2. Реализация

**Account ID:** `contact_id` или `company_id` из AmoCRM. Один контакт/компания → N сделок.

**Расчётные метрики:**

| Метрика | Формула | Описание |
|---------|---------|----------|
| Total Account Value | SUM(deal_value) по всем сделкам аккаунта | Общая ценность аккаунта |
| Expansion Revenue | Total Account Value − первая сделка | Доход от допродаж филиалов |
| Expansion Rate | COUNT(accounts с >1 deal) / COUNT(all accounts) | Доля расширяющихся аккаунтов |
| Net Revenue per Lead | Total Account Value / COUNT(leads по аккаунту) | Реальная ценность лида с учётом расширения |

### 8.3. Атрибуция

Канал атрибутируется по **первому лиду** аккаунта (first-touch). Expansion Revenue засчитывается тому же каналу.

Исключение: если второй лид имеет другой utm_source — фиксируем оба (first-touch + last-touch), но Expansion Revenue идёт первому каналу.

### 8.4. Отображение

На дашборде Pipeline & Velocity (Блок 4): таблица аккаунтов с total_deals > 1, sorted by Total Account Value.

Метрика «Net Revenue per Lead» добавляется в Channel Performance как дополнительная колонка.
